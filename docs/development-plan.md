# 五分钟地牢 — 开发计划文档

> 技术栈：HTML5 Canvas / 原生 JavaScript (ES Module) / Vite
> 美术风格：像素风
> 目标平台：移动端竖屏优先，PC 浏览器兼容
> 预估总工时：~243 小时 / 14 周

---

## 一、技术架构

### 1.1 游戏主循环 (Game Loop)

```
每帧执行流程：
  1. INPUT  — 采集输入快照（触控/键盘/鼠标）
  2. UPDATE — 固定时间步逻辑更新（60Hz，accumulator 模式）
  3. RENDER — 可变步长渲染（alpha 插值保证平滑）
```

关键实现：
- `maxDt` 钳制为 0.1s，防止切标签页后时间爆炸
- `deltaTime` 帧间隔时间，用于帧率无关运动
- `alpha` 用于逻辑步与渲染步之间的插值

### 1.2 简化 ECS 架构

```
Entity = { id, components: { Transform, Sprite, Health, ... } }
System.update(world, dt) → world.query('Transform', 'Health') → 批量处理
World = { entities Map, componentIndex: Map<string, Set<id>> }
```

简化原则：组件即 JS 对象，不用 class 继承，不做 archetype 优化。

### 1.3 碰撞检测

```
第1级：空间哈希网格（cellSize=64px）— 粗筛
第2级：圆形碰撞（distance < r1 + r2）/ AABB — 精确检测
```

### 1.4 对象池

子弹、敌人、粒子、浮动伤害数字、掉落物全部池化管理，减少 GC 压力。

### 1.5 状态机

```
BOOT → MENU → PLAYING ⇄ LEVEL_UP → RESULT → MENU
                  ↕ PAUSED
```

RunState 独立于场景状态，跨场景持久化单局数据（时间/分数/技能/击杀）。

### 1.6 移动端适配

- 虚拟摇杆：按在哪就在哪出现，非固定位置
- 逻辑分辨率固定 360x640，CSS 等比缩放
- `image-rendering: pixelated` 保持像素锐利
- DPR 适配：Canvas 实际像素 = 逻辑像素 x DPR

## 六、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| 性能不达标（低端手机<30fps） | 高 | 对象池从一开始实施；预留低画质模式；性能预算 500 drawCall/300 实体 |
| 数值平衡困难 | 高 | 所有数值 JSON 配置；记录每局数据驱动调整；参考吸血鬼幸存者曲线 |
| 美术资源不足 | 高 | 原型期全用代码占位；用开源素材(OpenGameArt/itch.io)；精灵 16x16/32x32 |
| 游戏缺乏"爽感" | 中 | 每个升级有明显视觉+数值变化；敌人密度足够高；Boss 战作为高潮 |
| 范围蔓延 | 高 | 严格执行 Phase 划分；技能硬上限 30 种；Must/Should/Could/Won't 优先级 |
| 移动端触控体验差 | 中 | 动态摇杆（按哪在哪出现）；摇杆区域限屏幕下半；死区 10px |
